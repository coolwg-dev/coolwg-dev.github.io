排序演算法比較：直覺、複雜度與實務選擇

📅 2026-04-01

第一部分 —— 為何排序依然關鍵

在需要處理大量資料的系統中，排序幾乎無所不在。資料庫使用排序來建立索引與加速查詢，分析流程會在分組與統計前對資料做排序，日常應用中的排行榜、搜尋結果與推薦清單也都依賴高效的排序結果。由於排序往往位於資料管線的關鍵路徑上，選擇不同演算法，會直接反映在延遲與資源消耗上。

像 Cormen、Leiserson、Rivest 與 Stein 合著的《Introduction to Algorithms》之所以把排序當作核心章節，是因為它很好地呈現了漸近複雜度、常數因子與實作細節之間的取捨。在日常開發中，我們通常不會從零開始寫排序程式碼，但理解各種演算法差異，能幫助我們更有根據地分析效能，並解讀標準程式庫排序函式的行為。

第二部分 —— 比較排序演算法的面向

若要有系統地比較不同排序演算法，可以從以下幾個面向著手：

  • 時間複雜度：在最好、平均與最壞情況下的漸近表現，例如 O(n log n) 或 O(n²)。
  • 空間使用：演算法是否屬於原地排序（只需常數額外記憶體），或是需要與輸入規模同階的輔助儲存空間。
  • 穩定性：對具有相同鍵值的元素，是否保留原本的相對順序，這在以多個欄位排序紀錄時格外重要。
  • 實作難度：是否容易正確實作、撰寫測試與除錯，是否容易踩到邊界條件或遞迴深度過深等問題。
  • 常數因子與快取行為：存取模式是否具備良好區域性、分支是否容易被 CPU 預測，這些因素在實際硬體上會帶來明顯差異。

即便兩個演算法在大 O 記號下擁有相同的時間複雜度，它們在真實負載上的表現仍可能差距很大。差異往往來自常數因子、快取命中率與資料分布等細節，因此除了紙上推導之外，針對實際或模擬資料做小型基準測試依然相當值得。

第三部分 —— 二次時間的經典演算法：泡沫、選擇與插入排序

在教學場合與小規模資料情境中，泡沫排序、選擇排序與插入排序是最常被介紹的三種演算法。它們在最壞情況下的時間複雜度皆為 O(n²)，但行為特徵並不完全相同。

  • 泡沫排序：反覆比較並交換相鄰且次序顛倒的元素，直到整個序列有序。內部迴圈會進行大量多餘的比較與交換，且快取友好性較差，因此除了示範用途，幾乎不會出現在生產系統。
  • 選擇排序：每一輪從尚未排序的部分找出最小元素，放到下一個輸出位置。比較次數與輸入順序無關，而交換次數僅為 O(n)。在「交換成本遠大於比較成本」時有一定吸引力，但 O(n²) 的比較次數仍限制了可處理的資料規模。
  • 插入排序：由左至右遍歷序列，將每個新元素插入到已排序前綴中的正確位置，並將較大的元素向右移動。最壞情況下時間複雜度為 O(n²)，但若輸入本身已大致有序，表現可接近 O(n)。許多高效實作會在子陣列很小時退回插入排序。

總結來說，這類二次時間演算法在理解概念與處理極小資料集方面非常有用，但並不適合大規模排序任務。

第四部分 —— O(n log n) 排序：合併排序、堆排序與快速排序

對於大型輸入，通用的比較式排序通常會採用 O(n log n) 的演算法，其中最常見的就是合併排序、堆排序與快速排序。

  • 合併排序：遞迴地把陣列一分為二，分別排序後再將兩個有序子序列合併。無論輸入分布為何，都能保證 O(n log n) 的時間，並且可以實作為穩定排序。主要缺點是在陣列情境中通常需要 O(n) 額外空間。
  • 堆排序：先建立二元堆，接著反覆取出堆頂元素並放到陣列尾端。堆排序在最壞情況下同樣提供 O(n log n) 時間，且可近似原地執行，只需極少額外記憶體。不過，其記憶體存取較為分散，快取友好性通常不如快速排序或合併排序。
  • 快速排序：選擇一個樞軸，把陣列分割為「小於樞軸」與「大於樞軸」兩部分，再遞迴排序。使用隨機化或三數取中等策略時，期望時間複雜度為 O(n log n)，且具有優良的區域性與快取表現。其最壞情況為 O(n²)，但現代實作往往採用 introsort 等技術，在遞迴過深時改用堆排序以避免退化。

這些 O(n log n) 演算法構成多數通用排序實作的基礎，不同標準程式庫多半在它們之間做出不同的混合與優化。

第五部分 —— 鎖定整數鍵的近似線性時間排序

當鍵值來自有限整數範圍，或可以拆解為固定寬度的「位元／數字」時，就可以使用非比較式排序，在合理假設下接近線性時間。

  • 計數排序：統計每個鍵值出現次數，再依序重建輸出。時間複雜度為 O(n + k)，其中 k 為鍵值範圍大小，額外記憶體亦為 O(k)。當範圍不大且資料分布適中時，非常高效。
  • 基數排序：逐位處理鍵值（例如由低位到高位），每一輪利用穩定排序（常見為計數排序）作為子程序。對於固定寬度的整數或長度受限的識別碼，基數排序在實務上往往能接近 O(n)。
  • 桶排序：依照鍵值區間將元素分配到不同桶中，再對各桶內部排序（經常採用插入排序），最後依序串接。效能對資料實際分布相當敏感。

這類演算法在處理大量整數、時間戳或結構化 ID 時特別有用，但適用範圍比通用比較式排序更窄，需要先確認前提是否成立。

第六部分 —— 標準程式庫在實務上的作法

現代語言的標準程式庫很少直接採用單一教科書演算法，而是偏好混合策略，在理論保證與實際表現之間取得平衡。舉例來說，Python 的 list.sort 以及 Java 中物件陣列的排序採用 Timsort，這是一種由 Tim Peters 設計、結合合併排序與插入排序的演算法，善於利用資料中已存在的有序片段。C++ 的 std::sort 通常實作為 introsort，也就是在快速排序基礎上，當遞迴過深時切換到堆排序。

這些混合演算法往往共享幾個原則：在子問題很小時採用插入排序；限制遞迴深度以避免快速排序的病態情況；偵測並善用部分有序輸入以減少工作量。理解這些模式，有助於說明為何標準程式庫排序往往比直接照抄教材演算法更快且更穩定。

結語

排序演算法之所以在計算機科學中長盛不衰，是因為它們濃縮了理論保證與實務表現之間的拉鋸。對多數日常開發情境而言，最安全的選擇幾乎永遠是使用語言本身提供的排序函式，並信任其背後經過多年打磨的混合演算法。當你需要更細緻地分析效能時，分辨二次時間演算法、O(n log n) 演算法與專門的近線性技術，能為溝通權衡與設計實驗提供清楚的語言。

若想更深入了解，可以參考《Introduction to Algorithms》與 Robert Sedgewick 等人的演算法著作，這些資料提供了對各種排序方法的嚴謹分析與實驗比較。基於實際業務資料建立小型基準測試，通常是判斷某個情境中適合採用哪一種排序演算法的最直接方式。
