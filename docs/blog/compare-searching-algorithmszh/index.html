比較常見搜尋演算法：取捨與實務指南

📅 2026-04-01

第一部分 —— 我們要解決的「搜尋問題」是什麼？

從抽象角度來看，搜尋就是回答「這批資料裡是否包含某個給定鍵？如果有，它在哪裡？」這樣的問題。在工程實務中，答案高度依賴資料的儲存方式：記憶體中的小型未排序清單、磁碟上的排序陣列，以及資料庫中的雜湊索引，行為都不一樣。Donald Knuth 的《The Art of Computer Programming, Volume 3: Sorting and Searching》、Cormen、Leiserson、Rivest、Stein 合著的《Introduction to Algorithms》，以及 Sedgewick 與 Wayne 的《Algorithms》等經典著作，都強調在選擇演算法前先把模型假設講清楚——資料是否有序、一次比較的成本多大、能否使用額外記憶體，以及資料更新是否頻繁。

時間複雜度提供了成本的第一近似。線性搜尋在最壞情況下需要檢查 n 個元素，記作 O(n)；在排序陣列上做二分搜尋，最壞情況則是 O(log n) 次比較。不過，常數因子、記憶體存取模式與 CPU 快取行為同樣重要。對非常小的集合來說，O(n) 與 O(log n) 之間的差異，往往遠小於函式呼叫或快取未命中的開銷。一個實用的思考方式是：在滿足資料模型前提的情況下，先採用最簡單的演算法，當量測結果顯示瓶頸時，再引入更複雜的結構。

第二部分 —— 線性搜尋：小規模或未排序資料的基線

線性搜尋（循序搜尋）會依序檢查每個元素，直到找到目標或走完整個集合。它適用於任何可迭代容器，無論是否有序，例如陣列、鏈結串列或資料串流。最壞時間複雜度為 O(n)，如果目標恰好在前面位置，最好情況可以接近 O(1)。

線性搜尋的優點在於簡單、穩健且幾乎沒有前置成本。它不要求資料預先排序，實作上也比較不容易出錯。對於規模不大的集合——幾十個，甚至一兩百個元素——在考慮分支預測與快取命中情況後，線性搜尋往往比更複雜的演算法更快。Knuth 在討論「小 n」 場景時就指出，當樣本規模很小時，只看漸進複雜度很容易被誤導。

當集合規模變大，或者必須頻繁重複搜尋時，線性掃描的限制就會顯現。如果你有成千上萬甚至上億筆紀錄，且幾乎每次請求都要搜尋，O(n) 的成本很快就會變得難以接受。此時通常需要藉由結構（排序加二分搜尋）或索引（雜湊表、平衡樹等），讓查找成本更接近常數時間或對數時間。

第三部分 —— 二分搜尋及其變體：面向有序資料

二分搜尋假設你可以隨機存取一個已排序的序列。在每一步，它會把目標與中間元素比較，然後丟棄一半搜尋區間。如此反覆折半，使得最壞時間複雜度為 O(log n)。在實際程式碼中，通常應優先使用標準函式庫提供的實作（例如 C++ 的 std::lower_bound、Python 的 bisect 或 Java 的 Collections.binarySearch），以避免手寫時常見的邊界與整數溢位錯誤。

關鍵前提在於底層資料結構必須維持排序。維護這個不變式是有成本的：在排序陣列中插入元素往往是 O(n)，因為可能需要搬移後續元素。對於以讀取為主、寫入相對少的情境——例如靜態設定表、編譯時產生的查找表或每月批次載入的參考資料——這樣的成本是可以接受的。對於寫入頻繁的情境，基於樹或雜湊的結構通常更合適。

二分搜尋也有若干變體，例如插值搜尋與指數搜尋。插值搜尋會根據鍵值的大致分布估算探測位置，在數值鍵近似均勻分布時，可能比傳統二分快；但在高度偏斜的資料上則會明顯退化。指數搜尋適用於事前不知道陣列上界的情況：它先以指數方式擴大搜尋區間，再在該區間內進行二分。這些演算法在《Introduction to Algorithms》等教材以及相關研究論文中有更完整的介紹，但在多數實際系統中，標準二分搜尋在複雜度與實作成本之間已經提供了很好的折衷。

第四部分 —— 雜湊表、樹結構與索引搜尋

當查找操作非常頻繁，且可以接受額外維護索引結構時，通常會從基於陣列的搜尋轉向內建索引的容器。雜湊表（map、dictionary）和平衡搜尋樹（例如紅黑樹、AVL 樹）是兩個典型選擇。此時你不再直接在陣列上掃描或二分，而是把「根據鍵找到位置」的責任交給容器。

雜湊表在期望上追求 O(1) 的查找、插入與刪除時間，透過對鍵計算雜湊值並映射到桶來實現。效能高度依賴雜湊函式品質與裝載因子；品質不佳的雜湊或惡意設計的鍵值會明顯拖慢操作並增加記憶體使用。相較之下，平衡樹維持鍵的有序性，並保證各種操作的時間複雜度為 O(log n)。它們天生支援區間查詢與有序遍歷（例如「所有介於 A 與 B 之間的鍵」），這是一般雜湊表較不擅長的。

資料庫與搜尋引擎進一步將這些概念發展為多層索引，例如 B-tree 與倒排索引。B-tree 透過在每個節點存放大量鍵來減少磁碟隨機存取，而倒排索引用「詞項 → 文件列表」的方式支援全文檢索。儘管這些系統比純記憶體容器複雜許多，其背後的權衡——讀寫成本、記憶體占用與存取模式之間的平衡——與我們在選擇線性搜尋、二分搜尋或雜湊查找時面對的是同一類問題。

第五部分 —— 實務中的選擇準則

  • 對極小規模集合（例如只有幾十項的設定清單），優先採用線性搜尋，保持簡單，並用量測來確認是否需要進一步優化。
  • 當資料自然保持排序且更新不頻繁時，可透過標準函式庫提供的二分搜尋介面，在不引入額外索引結構的前提下獲得很快的查找。
  • 當查找非常頻繁且伴隨大量插入或刪除時，應優先選用經過充分驗證的雜湊表或平衡樹容器，而不是自行設計索引方案。
  • 如果除了點查找之外，還需要有序遍歷或區間查詢，宜優先考慮樹結構而非雜湊表。
  • 在真實工作負載與具代表性的資料上進行效能分析。將來自 Knuth、Cormen 等教材以及 Sedgewick & Wayne 的演算法複雜度分析視為指引，但最終選擇應由你所在環境中的實測數據來決定。

透過同時參考演算法理論與實測結果，你可以在「能簡單就簡單」與「需要擴充時依然穩健」之間，找到合適的搜尋策略。
